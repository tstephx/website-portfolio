From Crashing Excel to Real-Time Automation

Replacing a fragile 55MB Excel file with SQL-based automation that increased Pinnacle fill rate from 57.9% to 94.1% — hitting 100% in July.

SQL / Redshift Haversine Formula Data Integration Salesforce

The Problem

The Pinnacle eligibility process — determining which delivery partners qualified for premium station assignments — ran on a 55+ MB Excel file with INDEX-MATCH calculations across 24,000 rows. It crashed regularly, used inconsistent traffic-dependent distance calculations, had quarterly data lag, and was maintained by a single person who built it years ago.

Pinnacle Launches (Before) 93 launches
Pinnacle Launches (Broken System) 42 launches

The results were predictable: Pinnacle launches dropped 55% (from 93 to 42) and 20 eligible candidates were bypassed entirely because the system couldn’t reliably identify them. A critical growth program was throttled by a spreadsheet that couldn’t handle its own data.

The Approach

I diagnosed root causes across three dimensions: data (inconsistent, lagging), process (single point of failure, no documentation), and tooling (Excel couldn’t scale). The solution needed to address all three simultaneously.

Three-Layer Solution

  • Data layer: Create a multi-source integration pipeline feeding fresh data into Amazon Redshift — eliminating quarterly lag.
  • Logic layer: Implement the Haversine distance formula in SQL for consistent geospatial calculations — eliminating traffic-dependent variability.
  • Delivery layer: Build an outreach prioritization algorithm connected to Salesforce — enabling automated, ranked candidate identification.

Key Decisions

Haversine Distance Formula

The original Excel file used driving distances from a mapping API, which varied based on traffic conditions. The same partner-to-station pair could show different distances depending on when you checked. I implemented the Haversine formula for great-circle distance calculations — consistent, reproducible, and fast. While driving distance is more accurate in absolute terms, consistency mattered more for eligibility ranking.

Haversine Formula
d = 2r · arcsin(√(sin²(Δφ/2) + cos(φ1) · cos(φ2) · sin²(Δλ/2)))

Great-circle distance between two points on a sphere — consistent, reproducible, traffic-independent

6-Step SQL Query Architecture

The transformation from raw data to prioritized outreach list required six sequential steps: (1) data extraction from multiple sources, (2) partner qualification filtering, (3) station requirement matching, (4) distance calculation via Haversine, (5) scoring and ranking, and (6) Salesforce-ready output formatting. Each step was a modular query that could be tested and maintained independently.

Multi-Source Data Integration

The Excel file had been updated quarterly by manually copying data from various systems. The new architecture pulled directly from source systems into Redshift on a daily refresh cycle. This eliminated both the data lag and the manual effort of maintaining the spreadsheet.

Salesforce Integration

The final step wasn’t just a list — it was an actionable queue in Salesforce with prioritization logic baked in. Recruiters could work through candidates in optimal order without needing to understand the underlying eligibility calculations.

Pinnacle fill rate jumped from 57.9% to 94.1% — a 36 percentage point increase — with July hitting 100%. The 20 eligible candidates who had been bypassed by the old system were identified and preserved. The crashing, quarterly-lagged Excel file was replaced with an automated, real-time system that required no manual maintenance.

Beyond the metrics, the system eliminated a critical single point of failure. The previous process depended on one person’s Excel expertise; the new process was documented, modular, and maintainable by anyone with SQL access.

Lessons Learned

  • Excel heroics are a sign of missing infrastructure. A 55MB file with 24,000-row INDEX-MATCH formulas is a system screaming for a database.
  • Consistency often beats accuracy. Haversine distances aren’t as “accurate” as driving distances, but they’re reproducible — and reproducibility is what eligibility decisions require.
  • Single points of failure aren’t just risk — they’re indicators. When a process depends on one person’s spreadsheet, that process is already broken.
